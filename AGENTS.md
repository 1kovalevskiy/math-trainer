# AGENTS.md

## Цель проекта
- `math-trainer` — TUI-приложение для тренировки математики у детей.
- Стек: Go + Bubble Tea.

## Структура проекта
- `cmd/main.go` — точка входа.
- `cmd/app` — сборка и запуск приложения.
- `internal/app` — весь код, связанный с Bubble Tea (экраны, `Model`, `Update`, `View`, обработка `tea.Msg`).
- `internal/controllers` — бизнес-логика и внешние вызовы, не завязанные на Bubble Tea.
- `internal/models` — доменные модели для контроллеров.
- `internal/providers` — абстракции/интерфейсы для внешних сервисов.
- `internal/storages` — абстракции/интерфейсы для БД.
- `internal/config` — инициализация конфига из YAML (+ ENV overrides).

## Правила по слоям
- `internal/app` не должен ходить в БД или внешние сервисы напрямую; только через контроллеры.
- `internal/controllers` не должен зависеть от Bubble Tea типов.
- Длительные операции запускаются через `tea.Cmd`, а результат возвращается в виде `tea.Msg`.
- Бизнес-правила находятся в контроллерах, а не в `Update`/`View`.
- `internal/models` содержит только модели предметной области и не должен тянуть UI-зависимости.

## Направление зависимостей
- Разрешено: `app -> controllers -> (providers, storages)`.
- Запрещено: обратные зависимости и циклические импорты.

## Конфигурация
- Основной источник: YAML-файл.
- Переменные окружения могут переопределять YAML.
- Все новые поля конфига добавлять в `internal/config` с тегами `yaml` и `env`.

## Кодстайл и практики
- Во все внешние вызовы передавать `context.Context`.
- Ошибки оборачивать через `%w` и поднимать вверх по слоям.
- Логирование вести структурированно через `slog`.
- Держать функции небольшими и явно разделять UI-логику и бизнес-логику.

## Тестирование
- В первую очередь покрывать тестами контроллеры.
- Для контроллеров использовать моки `providers` и `storages`.
- В тестах `internal/app` проверять переходы состояний по `tea.Msg`.
- Избегать реальных сетевых/БД вызовов в unit-тестах.
